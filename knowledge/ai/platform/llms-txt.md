# llms.txt

An open proposal for websites to provide "curated Markdown context" for LLMs. A `/llms.txt` (Markdown file) is placed at the site root, containing a curated set of links to content important for understanding the site, structured to be readable by humans, LLMs, and parsers alike. Proposed by Jeremy Howard (Answer.AI) on 2024-09-03. It addresses the problem that converting HTML (nav, ads, JS) into LLM-readable text is inaccurate, and that a whole site cannot fit in a context window — on the premise that "the site's author knows best," the author is asked to write a high-signal index themselves.

Spec at <https://llmstxt.org/>, repository is `AnswerDotAI/llms-txt` (Apache-2.0).

## Placement and file names

- **`/llms.txt`**: a Markdown file placed at the site root (e.g. `https://example.com/llms.txt`). It can also be placed at a subpath.
- **Markdown version of individual pages**: also provide clean Markdown by appending `.md` to the URL (e.g. `page.html.md`). For URLs without a filename, use `index.html.md`.
- **`llms-ctx.txt` / `llms-ctx-full.txt`**: context files generated by the official CLI `llms_txt2ctx` by expanding `llms.txt`. The former excludes the `Optional` section; the latter includes it.
- **`llms-full.txt`**: a **community/platform convention** (popularized by Mintlify and others) that inlines the full documentation into a single file. **Note that this is not part of the official spec.** While `llms.txt` is a lightweight index (~10–20 links), `llms-full.txt` carries the full text.

## File format

Markdown, but with a fixed structure intended for strict programmatic parsing. Order (**only the H1 is required**):

1. (Optional) BOM
2. **H1**: project/site name — **the only required element**
3. **blockquote (`>`)**: a short summary needed to understand the rest (recommended)
4. **Freeform Markdown**: any non-heading blocks (paragraphs, lists, etc.) supplementing background and notes
5. **H2 (`##`) sections**: each section contains a "file list"
6. **List items**: a Markdown list within each H2. Each item is a **required Markdown link `[name](url)`** plus an optional description following a `:`

### The special meaning of `## Optional`

An H2 heading that is literally `Optional` denotes a group of URLs that are **secondary and may be omitted when short context is needed**. The default generation of `llms_txt2ctx` excludes this; only the full generation includes it.

```markdown
# FastHTML

> FastHTML is a python library which brings together Starlette, Uvicorn,
> HTMX, and fastcore's `FT` "FastTags" into a library for creating
> server-rendered hypermedia applications.

Important notes:

- Although parts of its API are inspired by FastAPI, it is *not*
compatible with FastAPI syntax and is not targeted at creating API services

## Docs

- [FastHTML quick start](https://fastht.ml/docs/tutorials/quickstart_for_web_devs.html.md): A brief overview of many FastHTML features

## Examples

- [Todo list application](https://example.com/adv_app.py): Detailed walk-thru of a complete CRUD app

## Optional

- [Starlette full documentation](https://example.com/starlette-sml.md): Useful subset for FastHTML
```

The minimal valid file can be just a single H1 line: `# Project Name`.

## State of adoption (as of 2026-06)

**Major model provider crawlers do not automatically fetch `/llms.txt`.**

- Google has **explicitly stated it does not support** llms.txt (2025-07, Gary Illyes), taking the position that it has no SEO ranking value.
- Most AI crawlers such as GPTBot / ClaudeBot / PerplexityBot also crawl HTML directly and do not reference this file. Third-party surveys put adoption among public sites at roughly 10% (vendor-sourced figure, treat as a rough indicator).
- **Actual use is centered on on-demand fetching by IDEs / coding agents.** Cursor, Windsurf, Claude Code, GitHub Copilot, Cline, Aider, and others fetch it when a user explicitly points to a documentation URL. In other words, the consumption model is not "providers auto-discover it" but "agents explicitly reference it."
- Documentation platforms (Mintlify, GitBook, etc.) auto-generate `llms.txt` / `llms-full.txt`.

> Note: quantitative data on adoption rates and crawler behavior comes from SEO/analytics vendors, not from the providers themselves. The qualitative facts (Google's non-support, providers not auto-fetching, on-demand use by coding agents) are corroborated across multiple sources.

## Comparison with similar formats

| Target | Purpose | Difference from llms.txt |
|---|---|---|
| `robots.txt` | Access control over crawl permission | llms.txt does not control access; it is **content guidance** at inference time. Complementary |
| `sitemap.xml` | Exhaustive listing of all URLs (for search indexing) | llms.txt is a **curated, high-signal subset**. sitemap does not provide LLM-ready Markdown and is too voluminous |
| [[open-knowledge-format]] (OKF) | A knowledge library of typed conceptual documents | llms.txt is a **single index/signpost** directing to site content; OKF is a **structured knowledge bundle** with frontmatter |

llms.txt (the signpost at the entrance) / OKF (the library inside) / [[mcp-protocol]] (the librarian who fetches on request) are often organized as overlapping (non-competing) layers.
